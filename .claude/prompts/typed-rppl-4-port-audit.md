# Typed RPPL Graph — Continuation Prompt 4: Port Assignment Audit

## Background

A previous session added typed `inputs`/`outputs` ports to all 96 RPPL seeds (2 axioms, 5 capacities, 39 frameworks, 50 principles) in `app/src/data/rppl-seeds/`. The 39 frameworks and 50 principles were largely generated by an agent following heuristic rules — they compile, they have the right SHAPE, but they haven't been audited by a human for semantic correctness.

The port assignments affect:
- `verifyLifeGraph` — which frameworks the user's capacities must satisfy
- `encodeLifeGraph` — output labels in the compressed encoding
- Future recommendation logic — matching practices to the right capital outputs

Bad ports → bad recommendations and spurious gap warnings.

## Your job

Audit every seed's ports for semantic correctness. Fix mistakes. Fill in anything the agent missed or got generic.

## What a good port assignment looks like

**Inputs** — capacity prerequisites for this to work. Should reflect the actual cognitive/emotional prerequisites the framework demands.
- Stoicism requires: awareness (to observe reactions), care (to engage with virtue), agency (to choose response). ✓ Good.
- Permaculture requires: care (it's about tending), maybe awareness. Doesn't require humility or agency at the same level.

**Outputs** — what this produces when the person integrates it. Should be specific and domain-relevant.
- Stoicism outputs: emotional_regulation, resilience, spiritual (capital). ✓ Good.
- A body-health framework should output `living` capital, not `intellectual`. A decision framework should output `decision_clarity`, not `critical_thinking` (those are different).

**Bad patterns to look for**:
- Every framework gets `intellectual` as output → too generic
- Every capacity-requiring framework gets the same 3 capacity inputs → agent laziness
- Outputs that don't match the framework's actual domain (e.g., circadian biology outputting `intellectual` instead of `living`)
- Missing outputs for frameworks with clear capital effects (e.g., antifragility should mention `living`, austrian economics should mention `financial`)

## Concrete deliverables

### 1. Framework audit

For each framework seed in `frameworks.ts`:

1. Read the name, domain, description, and tenets
2. Check the `inputs` array: does every listed capacity genuinely need to be at `emerging+` for this to work?
3. Check the `outputs` array: do the outputs match what this framework actually produces? Are at least 1-2 outputs capital-typed? Are state outputs specific enough to be meaningful?
4. Fix mistakes; add missing outputs

Use this reference table for capacity prerequisites (from `capacities.ts:prerequisiteFor`):
- awareness → trivium, socratic-method, scientific-method, first-principles, stoicism, jungian-psychology, polyvagal-theory, nvc
- honesty → scientific-method, first-principles, holistic-management, jungian-psychology, general-semantics
- care → holistic-management, permaculture, stoicism, vedic-dharma-karma, flow-states
- agency → stoicism, non-aggression, first-principles, holistic-management, antifragility
- humility → trivium, scientific-method, first-principles, socratic-method, general-semantics, cognitive-biases

Frameworks in the capacity prerequisiteFor lists should have matching inputs. Other frameworks should have inputs that make domain sense (e.g., circadian biology needs awareness to notice timing, not necessarily humility).

### 2. Principle audit

For each principle seed in `principles.ts`:

1. Read name, domain, description, `axiom` one-liner
2. Check `outputs`: do they reflect what BELIEVING this principle produces? Not generic "intellectual capital" — specific downstream states.
3. Principles grouped by domain should feed their domain's capital (body principles → living; money principles → financial; etc.)
4. Fix mistakes

### 3. Cross-seed consistency

After per-seed audit, check global consistency:

- Every port `key` uses `lowercase_snake_case` (machine keys)
- Every `capital` port `key` matches one of: financial, material, living, social, intellectual, experiential, spiritual, cultural
- Every `capacity` port `key` matches one of: awareness, honesty, care, agency, humility
- No duplicate output ports within a single seed
- Each framework has at least one capacity input

### 4. Validate via `verifyLifeGraph`

Construct a test HumaContext with `capacityState` partially populated (e.g., awareness=developing, others=emerging). Run `verifyLifeGraph` against a set of patterns that reference various frameworks. Check that the reported `unsatisfiedInputs` match what you'd expect.

If the verification returns nonsensical blocking violations, it's because of bad input assignments. Trace back to the framework seed and correct.

### 5. Document your changes

After auditing, write a brief summary to `docs/rppl-port-audit.md` listing:
- Frameworks with major port corrections
- Principles with major port corrections
- Any systematic issues discovered (e.g., "agent assigned `intellectual` to every framework output; corrected to domain-specific capitals")

## What "done" looks like

A domain expert could read any 5 random seeds and agree the ports are correct. Running `verifyLifeGraph` on realistic user data produces suggestions that match what a thoughtful coach would surface. The port assignments stop being generic and start reflecting genuine RPPL expertise.

This is the unglamorous work that makes the whole system trustworthy. It's worth doing well.
