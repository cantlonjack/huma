# Prompt 02: Onboarding Time-to-Value — Show the Destination Early

## Objective
Reduce the number of conversation exchanges before the user sees what HUMA actually produces. Show a ghost/preview of the production sheet after the 2nd exchange — before decomposition is complete — so the user understands the destination and is motivated to continue.

## Why This Matters
Currently, a "seasonal project" aspiration requires 5-7 exchanges before decomposition. A "life system" requires 6-8. That's asking someone to trust a system they've never seen work. The voice bible says "respect the operator's intelligence" — but making someone answer 7 questions before showing them the product is a job interview, not respect.

The fix: after exchange 2, show a ghost preview of what the production sheet *will* look like. "Here's roughly where this is headed. Let me ask two more things to make it specific."

## Architecture

### New Component: `SheetPreview`

Location: `app/src/components/onboarding/SheetPreview.tsx`

This is a simplified, read-only preview of what a production sheet looks like, using the information gathered so far. It should:

1. **Appear after the 2nd user message** in the `/start` conversation flow
2. **Show 2-3 placeholder entries** based on what HUMA knows so far
3. **Be visually distinct from the real sheet** — use reduced opacity (0.6), dashed borders, and a label like "Where this is headed"
4. **Update progressively** as more context arrives (exchanges 3, 4, 5 refine it)
5. **Transition to the real sheet** after decomposition completes

Visual treatment:
- Same card layout as `CompiledEntryRow` (serif headline, sans detail, dimension tags)
- But: dashed border (border-dashed), opacity-60, sand-100 background instead of white
- Skeleton shimmer on fields that aren't yet populated
- Header text: "Your day will look something like this" (Cormorant Garamond, 15px)
- Subtext: "A few more questions and I'll make it specific." (Source Sans 3, 13px, earth-400)

### Generation Strategy

Do NOT call Claude for the preview. Generate it client-side from available data:

1. **After exchange 2**: Use the aspiration raw text + any extracted context to build 2 generic-but-relevant entries. For example, if the user said "I want to get my garden producing food," generate:
   - Entry 1: "Plan the spring beds" / detail: shimmer / dimensions: [Home, Body]
   - Entry 2: "Check what seeds you have" / detail: shimmer / dimensions: [Home, Money]

2. **After exchange 3-4**: Replace shimmer with actual specifics from context (e.g., if they mentioned raised beds, update to "Map out the raised bed layout").

3. **After decomposition**: Fade out the preview, show the real decomposition result.

Implementation approach in `useStart.ts`:
- Track `messageCount` (user messages only, not HUMA responses)
- When `messageCount >= 2` AND `decomposedBehaviors` is empty (haven't decomposed yet), set `showSheetPreview = true`
- Build preview entries from: `knownContext`, `pendingAspiration.rawText`, any `recentDimensions`
- The preview entries are generated by a pure function `buildSheetPreview(rawText: string, context: Partial<HumaContext>, dimensions: string[]): PreviewEntry[]` in a new file `app/src/lib/sheet-preview.ts`

### Preview Entry Type

```typescript
type PreviewEntry = {
  headline: string;        // 4-8 words, may contain shimmer placeholder
  detail: string | null;   // null = show shimmer
  dimensions: string[];    // from context or inferred from aspiration text
  isPlaceholder: boolean;  // true = dashed styling
};
```

### `buildSheetPreview` Logic

This function uses keyword matching against the aspiration text and known context to generate plausible preview entries. It does NOT call Claude.

```
Input: "I want to get my garden producing food"
Context: { place: { property: "2 acres" } }
Output: [
  { headline: "Sketch the growing plan", detail: null, dimensions: ["home", "body"], isPlaceholder: true },
  { headline: "Inventory seeds and starts", detail: null, dimensions: ["home", "money"], isPlaceholder: true },
  { headline: "Prep the first bed", detail: null, dimensions: ["home", "body"], isPlaceholder: true },
]
```

Use the decomposition templates in `app/src/data/decomposition-templates.ts` (58KB) as a source — they already map aspiration keywords to behaviors and dimensions. Match the user's raw text against template keywords and pull 2-3 relevant `thisWeek` behaviors.

### Integration into `/start` Page

In `app/src/app/start/page.tsx`, the conversation thread currently shows messages in sequence. Insert the `SheetPreview` component:

- **Position**: Below the latest HUMA message, above the input field
- **Trigger**: `messageCount >= 2 && !decomposedBehaviors.length`
- **Animation**: Fade in with the standard easing curve (`cubic-bezier(0.22, 1, 0.36, 1)`), 400ms duration
- **Dismiss**: Automatically fades out when decomposition arrives (replaced by the real DecompositionPreview)

### Changes to Conversation Flow

The conversation architecture stays the same — HUMA still asks 3-7 questions depending on complexity. But the *perceived* time-to-value drops dramatically because the user can see what they're building toward.

Add a one-line contextual message from HUMA after the preview appears (injected client-side, not from Claude):
> "Here's roughly where this is headed. Let me ask a couple more things to make it yours."

This message should be styled as a `contextNote` (italic, earth-400 text) — the same styling used for system notes in the current conversation thread.

## Design Constraints
- Follow the design system: sand-50 background, Cormorant Garamond headlines, Source Sans 3 body
- Dashed borders use `border-dashed border-sand-300`
- No shadows, no gradients, no elevation — just opacity and border style to distinguish from real content
- Respect `prefers-reduced-motion` — if reduced motion, skip the fade-in
- Mobile-first: preview should be full-width on mobile, max-w-[680px] on desktop
