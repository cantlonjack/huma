## What HUMA Is

HUMA is a way of seeing. A perceptual grammar for practical wisdom.

Like literacy made written knowledge accessible, HUMA makes the systemic nature of individual human life visible -- the connections between dimensions, the leverage points, the patterns that work across contexts. There is a moment when someone stops seeing isolated problems and starts seeing one connected system. HUMA exists to produce that moment. At scale. For everyone.

**Nine-word tagline:** See the whole. Find the leverage. Practice what works.

**Two layers:**
- **Application layer** (for-profit product) -- three modes: Design, Operate, Evolve. This makes money.
- **Protocol layer** (RPPL -- Reality Pattern Protocol Layer, open standard) -- structured, contextual, validated, connected, living knowledge. This makes history.

The application makes money. The protocol makes history.

## The Naming Hierarchy

Four names, four contexts. Never confused.

- **HUMA** -- what users know. The app, the game, the practice. "I'm learning HUMA." The only name that ever faces the user.
- **RPPL (Reality Pattern Protocol Layer)** -- the open technical protocol. Developer-facing only. Describes the notation system, pattern schema, dimensional framework, compilation rules.
- **Pattern literacy** -- the capacity HUMA develops. Used in educational contexts. Parallel to literacy and numeracy.
- **Perceptual grammar** -- the theoretical description. Used in foundational and philosophical contexts. A language you see with, not one you speak.

### Progressive Depth

_Canonical spec: HUMA_PRODUCT_SURFACE.md §04_

Never forced. Always invited.

1. **Lotus Flow** (Day 1, 90 sec) -- 11 screens: entity type, life stage, governance, 8-capital spectrum, archetype synthesis, WHOLE evolution. First pattern recommendation.
2. **Context Deepening** (Day 2+, at operator's pace) -- Ikigai, Purpose, Vision. Each petal is 5-15 min. Each makes compiled recommendations sharper. Each evolves the WHOLE.
3. **One-Thing Card** (when context is deep enough) -- requires Lotus Flow + one context petal. Socratic framing, Got it / Not today (equal weight)
4. **Behavior + Design** -- pattern mapping, enterprise compilation, validation. The workspace becomes the operator's project.
5. **Full Depth** (Ongoing) -- masterplan, seasonal review, pattern contribution. Three-tab experience.

### Warmth System

_Canonical spec: HUMA_DESIGN_SYSTEM.md §05.5_

Visual states that communicate testing status without labels or numbers:

- **Faint** (~25% opacity) -- untested aspiration
- **Emerging** (~55% opacity) -- being tried, first weeks
- **Solid** (100% opacity, sage fill) -- validated, 4+ consecutive weeks meeting target
- **Shifting** (~80% opacity, amber tint) -- being revised based on learning
- **Faded** (~15% opacity, dashed border) -- tried, did not work, archived with history

## The 6-Phase Map Conversation (earned at week 2-3)

_Canonical spec: HUMA_PRODUCT_SURFACE.md §07-§12, HUMA_TECHNICAL_SPECIFICATION.md §04_

1. **Ikigai** -- Who are you? Warm, curious. Ends with Essence synthesis.

4. **Enterprise Map** -- What could you build? Real numbers, tested against QoL.
5. **Nodal Interventions** -- Where do you begin? Cascade chains.
6. **Operational Design** -- What does your week look like? Rhythm, validation, seasonal arc.

### QoL Decomposition Chain

```
"What matters" statement ("Evenings free for my daughter")
  -> What makes it work (all work done by 3pm)
    -> Weekly commitments (Tue/Thu: pack days done by 2pm)
      -> Daily behaviors (today's schedule with hard stop)
        -> Validation (workspace feedback + weekly insight)
          -> Adjustment (systemic, never personal)
```

When validation fails, HUMA ALWAYS looks at the system, NEVER at the person.

## Sovereignty Principles

_Canonical spec: HUMA_VISION_AND_STRATEGY.md §05, HUMA_ETHICAL_FRAMEWORK.md §02_

Every thinker in HUMA's lineage built systems that make individuals more sovereign, not less.

- **Data stays with the operator.** Local-first where possible. HUMA is custodian, not owner.
- **The protocol is open.** If HUMA the company disappears, the language survives. Anyone can build a compatible application.
- **AI serves, not decides.** Three roles only: mirror (shows patterns you haven't seen), library (retrieves relevant patterns), translator (adapts patterns to your context).
- **No external data harvesting.** No bank connections, no calendar sync, no health app integration.
- **The graduation metric.** Success = the operator no longer needs the tool. Measured passively, never displayed as a score.

### What HUMA Does NOT Build

1. A system that harvests data to create a model the user does not control
2. A system that tells people what to do (the one-thing card suggests, never prescribes)
3. A system that creates dependency through personalization (lock-in)
4. A system that requires a company to mediate between people and wisdom
5. A system where AI replaces human judgment instead of developing it
